# AlphaPilot PRD

> 产品名称：AlphaPilot  
> 产品定位：面向数字货币市场的 AI 自主交易系统  
> 当前范围：仅 Binance、仅数字货币、MVP 优先 BTC / ETH  
> 核心目标：在严格风控和受限策略框架下，实现 AI 自主交易，并通过经验沉淀和受控学习持续优化决策质量

---

# 1. 产品背景

当前市面上很多“AI 交易产品”主要停留在以下层面：

- 行情分析
- 信号提示
- 策略建议
- 聊天式问答

它们通常存在几个问题：

- 无法形成真正的交易闭环
- AI 输出缺少结构化约束
- 风控体系独立性不足
- 无法持续沉淀经验形成系统能力
- 过度依赖人工执行

AlphaPilot 希望解决这些问题，构建一个真正可运行的 AI 交易系统。

---

# 2. 产品目标

## 2.1 MVP 目标
在 Binance 上实现一个最小可运行的自主交易闭环：

1. 获取市场数据与账户状态
2. 计算指标与特征
3. 识别市场状态
4. AI 输出结构化决策
5. 风控校验是否允许执行
6. 自动执行交易
7. 监控持仓与收益
8. 记录经验并生成复盘

## 2.2 中期目标
在 MVP 稳定运行后，逐步增强：

- 策略评分器
- 参数优化器
- 经验检索增强
- Shadow Mode 验证
- 自动回滚机制
- 市场状态适应性优化
- AI Trader / Program Trader 双模式
- 多智能体职责分工（Decision / Prompt / Review / Ops / Factor）
- 交易归因分析（Attribution）
- 因子智能 / 因子库方向
- Web 控制台 + 消息端提醒/诊断的协同体验

## 2.3 长期目标
把 AlphaPilot 打造成一个：

- 可自主交易
- 可持续学习
- 可被审计
- 可严格控风险
- 可扩展到多策略和多币种

的 AI 交易平台。

# 3. 产品定位

AlphaPilot 的定位不是“交易建议助手”，而是：

**一个在明确边界内可自主运行的 AI 数字货币交易系统。**

它的价值来自四个关键词：

- 自主决策
- 风控约束
- 执行闭环
- 受控进化

---

# 4. 核心原则

## 4.1 AI 可以自主，但不能越权
AI 可以输出交易动作，但不能直接绕过风控执行。

## 4.2 风控优先级高于 AI
模型输出再强，也必须服从硬风控规则。

## 4.3 所有输出必须结构化
交易决策必须使用固定 schema 输出，便于验证、审计和执行。

## 4.4 所有学习必须可验证
任何参数调整、策略权重切换或学习结果，都必须经过验证后才能生效。

## 4.5 所有关键操作必须可回滚
系统必须支持参数回滚、策略降级、自动停机和人工干预。

# 7. 核心使用场景

## 7.1 自动交易运行场景
系统按固定周期运行：

- 拉取市场数据
- 拉取账户状态
- 计算指标与特征
- 识别市场状态
- AI 生成决策
- 风控审核
- 自动执行
- 记录结果

## 7.2 持仓管理场景
当存在仓位时，系统自动判断：

- 继续持有
- 部分减仓
- 止盈
- 止损
- 风险保护性平仓

## 7.3 风险异常场景
遇到以下情况时系统自动触发保护：

- 日内亏损超限
- 连续亏损超限
- API 异常
- 市场波动异常
- 滑点异常
- 数据异常

## 7.4 复盘与学习场景
每日或每个学习周期结束后：

- 汇总交易结果
- 提取经验
- 评估策略效果
- 更新候选参数
- 验证后决定是否切换

# 8. 产品演进方向（新增采纳）

## 8.1 已采纳的中短期方向

基于对外部项目的参考与当前 AlphaPilot 路线，确认采纳以下方向：

### 8.1.1 AI Trader / Program Trader 双模式
- AI Trader：适合依赖市场理解、综合判断、解释性较强的策略
- Program Trader：适合固定规则、技术指标驱动、可回测验证的策略
- 当前阶段先保留 AI Trader 为主线，Program Trader 作为后续扩展方向

### 8.1.2 多智能体职责分工
后续产品能力按职责拆分，而不是单一 AI 包办全部能力：
- Decision AI
- Prompt AI
- Review / Attribution AI
- Ops / Diagnose AI
- Factor AI

### 8.1.3 交易归因分析（Attribution）
后续分析系统需要支持按以下维度拆解盈亏与表现：
- 币种
- 时间段
- 触发类型
- 因子/信号来源

### 8.1.4 因子智能 / 因子库方向
在现有指标与市场特征基础上，逐步演进到：
- 因子库
- 因子评估
- 因子触发
- 因子归因
- 因子辅助 AI 决策

### 8.1.5 Web 控制台 + 消息端协同
产品体验上采用：
- Web 控制台负责主操作与管理
- 消息端（Telegram / Discord 等）负责提醒、诊断、轻交互
- AI 负责配置引导、诊断与解释

## 8.2 当前明确不纳入近期范围

以下方向暂不纳入近期规划：
- 多交易所支持
- 多模型同台 PK / 竞技场式比较

# 9. MVP 功能拆解

## 9.1 P0 功能

### 9.1.1 Binance 行情与账户接入
#### 目标
接入 Binance API，获取行情、账户、持仓和订单数据。

#### 功能
- 实时价格获取
- K 线数据获取
- 成交量获取
- 账户余额获取
- 当前持仓获取
- 当前订单与成交信息获取

#### MVP 范围
- 交易对：BTCUSDT、ETHUSDT
- 周期：15m、1h

---

### 9.1.2 技术指标与特征计算
#### 目标
统一计算 AI 决策和风控所需的市场特征。

#### 指标
- EMA20 / EMA50 / EMA200
- RSI
- MACD
- ATR
- 布林带
- 成交量均值
- 波动率
- 高低点突破
- 支撑阻力摘要

#### 输出要求
- 结构统一
- 可缓存
- 可复算
- 不依赖 LLM 内部推算

---

### 9.1.3 市场状态识别
#### 目标
识别当前市场环境，为策略选择和风险级别提供输入。

#### 状态分类
- trending_up
- trending_down
- ranging
- chaotic

#### 输入特征
- 均线排列
- 波动率
- ATR
- 成交量变化
- 回撤频率
- 趋势强度

---

### 9.1.4 AI 结构化交易决策
#### 目标
在给定上下文中输出固定 schema 的交易决策。

#### 输入
- 市场快照
- 技术指标
- 当前持仓
- 账户状态
- 市场状态
- 风控状态
- 相似经验摘要
- 当前可用策略框架

#### 输出动作
- OPEN_LONG
- CLOSE_LONG
- HOLD

#### 输出字段
- symbol
- timeframe
- action
- confidence
- entry_type
- entry_price
- stop_loss
- take_profit
- position_size_pct
- strategy_mode
- reasoning
- risk_note

#### 兜底机制
以下情况统一回退为 `HOLD`：

- JSON 解析失败
- 字段缺失
- 止损缺失
- 仓位比例异常
- 动作非法
- 系统处于熔断状态

---

### 9.1.5 受限策略框架
#### 目标
限制 AI 决策只在白名单策略内进行。

#### MVP 预置策略
1. 趋势跟随
2. 突破确认
3. 观望模式

#### 策略作用
- 限制 AI 行为空间
- 提高可解释性
- 便于回测与验证
- 降低幻觉式下单风险

---

### 9.1.6 执行守卫
#### 目标
所有交易动作都必须经过风控和规则引擎的批准。

#### 检查项
- 是否已有同方向持仓
- 是否超出最大仓位
- 是否缺失止损
- 止损距离是否异常
- 风险收益比是否达标
- 是否处于熔断状态
- API 是否正常
- 市场是否异常

#### 输出结果
- PASS
- REJECT
- DEGRADE

---

### 9.1.7 自动下单与自动平仓
#### 目标
在执行守卫通过后，自动完成下单和平仓。

#### 功能
- 市价下单
- 限价下单
- 自动挂止损
- 自动挂止盈
- 自动平仓
- 异常重试
- 状态回写
- 重复下单保护

---

### 9.1.8 基础风控熔断
#### 目标
控制亏损扩张，防止系统连续失控。

#### 核心规则
- 单笔最大风险限制
- 最大持仓比例限制
- 每日最大亏损限制
- 连续亏损熔断
- API 异常暂停
- 数据异常暂停
- 人工一键停机

#### 默认建议值
- 单笔风险：0.5% ~ 1%
- 最大单币持仓：10% ~ 20%
- 日亏损上限：2% ~ 3%
- 连续亏损熔断：3 笔

---

### 9.1.9 持仓与收益监控
#### 目标
实时展示系统运行状态和收益状态。

#### 展示内容
- 当前账户权益
- 当前持仓
- 浮动盈亏
- 累计收益
- 今日收益
- 当前风控状态
- 当前策略状态
- 当前市场状态

---

### 9.1.10 交易日志与复盘解释
#### 目标
记录完整交易链路，并自动生成复盘信息。

#### 记录内容
- 决策时间
- 市场快照
- AI 输出
- 守卫判定
- 订单执行结果
- 平仓结果
- 最终盈亏
- 风控事件
- 决策理由

#### 输出结果
- 当日交易次数
- 胜率
- 总收益
- 最大单笔亏损
- 最大回撤
- 典型盈利/亏损案例

### 9.1.11 经验库（基础版）
#### 目标

#### 记录字段

# 10. 自我进化系统设计

## 10.1 目标
让系统在不突破硬风控边界的前提下，基于历史交易结果持续优化：

- 策略选择
- 参数配置
- 市场状态识别
- 仓位建议
- 经验摘要注入

## 10.2 原则
- 只允许受控进化
- 硬风控不可学习
- 所有学习必须先验证
- 所有变更必须可回滚
- 学习失败时默认降级为更保守模式

## 10.3 核心模块

### 交易经验库
沉淀每笔交易上下文和结果。

### 策略评估器
持续评估不同策略在不同环境下的表现。

### 参数优化器
在白名单范围内产生候选参数。

### 市场状态识别器
持续优化 regime 识别阈值。

### 学习控制器
控制学习节奏、灰度发布和自动回滚。

## 10.4 学习闭环
1. 收集最近 N 笔交易和市场快照
2. 更新经验库
3. 评估策略表现
4. 生成候选更新
5. 历史验证
6. Shadow Mode 验证
7. 小流量上线
8. 表现变差则自动回滚

## 10.5 不可学习项
- 最大单笔风险上限
- 最大总仓位上限
- 每日最大亏损阈值
- 熔断规则
- 必须止损规则
- API 异常停机规则

## 10.6 可学习项
- 策略权重
- 指标参数
- 币种偏好
- 市场状态阈值
- 仓位建议系数
- 提示词中的经验摘要

# 13. 非功能要求

## 13.1 可解释性
每一笔交易和每一次学习都必须可解释。

## 13.2 可审计性
所有关键输入、输出、校验、执行和学习结果必须留痕。

## 13.3 安全性
- API Key 加密存储
- 权限隔离
- 幂等执行
- 风控优先级高于模型

## 13.4 稳定性
- Binance API 重试
- WebSocket 断线重连
- LLM 调用失败兜底
- 指标计算失败兜底
- 熔断状态优先级最高

## 13.5 幂等性
- 下单使用 trace_id
- 重试不重复下单
- 订单状态可恢复
- 止损止盈逻辑可恢复
