### 4) `/admin/users`
- 页面路由：`GET https://www.danerlt.top/ap-dev/admin/users` → `200 OK`
- 页面 JS chunk 已部署：`/app/admin/users/page-*.js`
- 数据接口：`GET /ap-dev/api/admin/users`（带 bearer token）→ `200 OK`
- 返回数据：当前至少包含默认管理员用户 1 条
- 结合源码核对：
  - 页面会调用 `GET /admin/users`
  - 页面内提供 role/status 下拉，并通过 `PATCH /admin/users/{id}` 更新
  - 成功反馈文案：`已更新用户 #${userId}`
  - 失败反馈文案：`更新用户失败` / `加载用户列表失败`
- 结论：**通过（读路径通过，更新反馈逻辑已实现；本轮未实际 PATCH）**

## 风险 / 限制
1. **缺少真实浏览器交互验证**
   - 本轮无法直接完成 DOM 级点击、表单输入、toast/提示可视化确认。
   - 目前结论对“后端真实可用性 + 前端代码路径已部署”可信，但对“视觉交互是否完全正常”仍不如浏览器实测。
2. **admin users / symbols 写操作未实测**
   - 为避免污染 dev 数据，本轮没有执行真实 `PATCH /admin/users/{id}` / `POST|PATCH /admin/symbols`。
   - 当前只验证到：接口在线、页面代码已接好、成功/失败反馈逻辑存在。
3. **默认管理员来源依赖示例文件与线上行为交叉验证**
   - 我没有读取真实 `.env.dev-server`。
   - 但示例默认管理员与线上登录成功结果一致，说明 dev 实际部署大概率已正确引导。

## 优先级建议
- **P1：补一轮真实浏览器烟测**
  - 尤其是登录跳转、admin 页面 hydration、移动端布局、表单提交后的可视反馈。

- **P2：给 admin 写操作补可观测回执**
