# 2026-03-17 Agent Orchestration Design

## 做了什么

1. 将用户原始“多 bot”想法收敛为四角色模型：
   - alpha-pilot
   - planner
   - developer
   - ops
2. 明确群内只汇报阶段性结果，而不是过程性碎片
3. 明确每个子角色有自己的上下文，但长期状态落地为 artifacts，避免上下文爆炸
4. 设计 feature / bug 两类工作流草案
5. 设计 run artifacts、handoff schema、input whitelist、broadcast protocol
6. 为后续 Telegram 多 bot 接入预留配置结构
7. 根据最新要求，进一步收敛主流程为：planner 先分析需求并更新 docs，developer 再实现并补齐白盒测试/单元测试，最后由 ops 验收和部署

## 为什么这样做

相比大量职责 bot：
- 四角色模型更容易控制 token 消耗
- 群里更干净
- 工程复杂度更低
- 仍然保留明确分工

ops 合并验收、测试结果复核、review、PR、merge、deploy，但内部再拆成短子阶段，以避免成为一个过胖上下文桶。

同时将质量责任前移：developer 不是“只写功能代码”，而是必须连同白盒测试和单元测试一起交付，ops 负责最终验收与部署把关。
