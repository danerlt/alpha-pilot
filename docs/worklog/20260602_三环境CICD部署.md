# 三环境 CI/CD 部署体系（2026-06-02）

## 做了什么

按 `dev → uat → main(prod)` 三环境 GitFlow，搭建 push 即自动部署的 CI/CD。

### 分支模型
- `dev` / `uat` / `main` 三长存分支（均从 main 切），`feat-xxx` 从 dev 切，先合 dev。
- 流转：feat→dev（部署 dev）→ dev→uat（部署 uat）→ uat→main（部署 prod）。

### 新增/改动文件
| 文件 | 内容 |
|------|------|
| `.gitignore` | `workflows/` → `/workflows/`（根锚点），放行 `.github/workflows/` |
| `.github/workflows/_deploy.yml` | 可复用部署 workflow（SSH 进服务器跑 deploy 脚本，按环境选 clone 目录，绑 GitHub Environment） |
| `.github/workflows/deploy-dev.yml` | push `dev` → 部署 dev |
| `.github/workflows/deploy-uat.yml` | push `uat` → 部署 uat |
| `.github/workflows/deploy-prod.yml` | push `main` → 部署 prod（environment: prod 审批门） |
| `docker/docker-compose.uat.yml` | 新建 UAT 栈（ap-uat，8002/3002，pg 5434 / redis 6381，db alphapilot_uat，ROOT_PATH /ap-uat） |
| `docker/nginx/alpha-pilot.conf` | `/ap-test` 段重命名为 `/ap-uat`（前一 commit） |
| `scripts/deploy-dev.sh` | `git pull origin main` → `git pull origin dev` |
| `scripts/deploy-uat.sh` | 新建（拉 uat，compose uat，端口 8002，/ap-uat） |
| `scripts/deploy-prod.sh` | 交互确认加 TTY 判断 + `FORCE_DEPLOY=1`，CI 无 TTY 自动跳过 |
| `docs/deploy-ci.md` | 部署指南（分支模型 / Secrets / 审批门 / 服务器准备 / 流程速查） |
| `CLAUDE.md` | 新增"分支模型与 CI/CD"节，自动部署规则改为 CI 触发 |

### 关键设计
- 三环境共用一台服务器、各自独立 clone（dev/uat/main 分支），CI 用 `DEPLOY_DIR_{DEV,UAT,PROD}` 区分目录。
- SSH 凭证一组共用：`DEPLOY_SSH_{HOST,USER,KEY,PORT}`。
- prod 用 GitHub Environment `prod` + required reviewer 做人工审批门（接 mainnet）。

## 为什么
- 原 `deploy-dev.sh` 写死 `git pull origin main`，与"dev 分支部署 dev"矛盾。
- 原 `deploy-prod.sh` 的 `read -p` 交互确认在 CI 无 TTY 会卡死。
- `.github/workflows/` 被 `.gitignore` 的 `workflows/` 误伤，CI 文件进不了 git。

## 如何验证
- 全部 workflow / compose YAML `yaml.safe_load` 通过。
- 三个 deploy 脚本 `bash -n` 语法通过。
- `.github/workflows/` 已脱离 gitignore（`git check-ignore` 确认）。
