# 2026-03-17 Agent Orchestration Stage Validation

## What changed

补了一步低风险但更可执行的 orchestration glue：

1. 强化 `scripts/run_agent_stage.py`
   - 增加角色归一化，兼容 `developer` / `implementer` 历史命名差异
   - 增加阶段校验，按 `state.currentStage` 限制 planner / developer / ops 的可执行时机
   - 增加上游工件校验，developer / ops 在关键 handoff 或产物缺失时会直接失败退出
   - 增加模板占位内容校验，避免把 `REPLACE_ME` / `RUN_TEMPLATE` 之类的模板内容当成真实交付继续往下游传
   - prompt 中补充当前状态、scope、handoff 内容，让下游执行输入更完整

2. 更新文档
   - `docs/agent-orchestration/运行器使用说明.md`
   - `docs/agent-orchestration/最小胶水脚本.md`

## Why

当前仓库已经有 run 初始化、state 更新、handoff 写入、public status 渲染这些脚手架，但 `run_agent_stage.py` 之前还是“只会吐 prompt 和命令”。

这会带来两个实际问题：
- 阶段错位时也能继续往下跑，容易把 run 状态和真实执行脱节
- 模板文件已经存在于 `RUN_TEMPLATE` 中，如果不做校验，下游很容易误把占位内容当成真实 plan / implement 结果

所以这一步选择补“执行前置校验”，是从最小脚手架往可执行层推进的一个合适小步，而且不影响现有服务运行。

## Verification

在本地做了 3 组验证：

1. 新建 run 后，planner 可正常生成 prompt 与 runner command
   - 命令：`python3 scripts/init_agent_run.py ...`
   - 命令：`python3 scripts/run_agent_stage.py R-20260317-002 planner`
   - 结果：成功输出 `role=planner`、prompt 文件路径、runner command

2. 新建 run 仍处于 `intake` 时，developer 会被阶段校验拦住
   - 命令：`python3 scripts/run_agent_stage.py R-20260317-002 developer`
   - 结果：报错 `Role 'developer' cannot run at stage 'intake'`

3. 将同一 run 手动推进到 `implement` 后，developer 会被占位 handoff 校验拦住
   - 命令：`python3 scripts/update_agent_state.py R-20260317-002 --status running --stage implement`
   - 命令：`python3 scripts/run_agent_stage.py R-20260317-002 developer`
   - 结果：报错 `placeholder artifacts detected: handoff/planner-to-implementer.yaml`

4. 已完成 run 若当前阶段是 `done`，ops 不能被误触发
   - 命令：`python3 scripts/run_agent_stage.py R-20260317-001 ops`
   - 结果：报错 `Role 'ops' cannot run at stage 'done'`
