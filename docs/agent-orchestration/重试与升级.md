# Retry 与升级策略

## 目标

让系统默认能自动修一轮，但不会无限循环。

## 基本规则

1. planner 默认不自动重试业务决策分歧，直接请求拍板
2. developer 可根据 ops 的问题摘要自动修复
3. ops 发现实现问题时，优先结构化打回 developer
4. 达到最大重试次数后，升级给 alpha-pilot 和老板

## Ops -> Developer 打回格式

```yaml
handoff:
  runId: R-EXAMPLE
  from: ops
  to: developer
  stage: ops-verify-and-ship
  status: retry
  summary: 验证未通过，需要修复以下问题
  issues:
    - 空输入边界条件未覆盖
    - 一个回归测试失败
  constraints:
    - 不扩大修复范围
  nextAction: patch implementation and resubmit
```

## 最大重试建议

- feature implement: 2 次
- bug fix: 2 次
- ops verify-and-ship: 2 次

## 升级条件

满足任一条件时升级：
- 同一问题重复出现
- 需要改变原定方案
- 需要越过风险边界
- staging deploy 连续失败

## 升级播报

由 alpha-pilot 向群里播报：
- 当前阻塞点
- 已尝试过什么
- 需要老板拍板什么
