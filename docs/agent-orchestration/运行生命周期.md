# Run 生命周期

## 目标

定义 alpha-pilot 如何创建、推进、阻塞、完成一个 run。

## 生命周期阶段

1. `intake`
2. `plan`
3. `implement`
4. `ops-verify-and-ship`
5. `summary`
6. `done`

Bug 流对应：
1. `intake`
2. `bug-plan`
3. `fix`
4. `ops-verify-and-ship`
5. `summary`
6. `done`

## 创建 run

当 alpha-pilot 收到新任务时：

1. 生成 run id
2. 复制 `runs/RUN_TEMPLATE`
3. 写入：
   - `task.md`
   - `state.json`
4. 根据任务类型选择 workflow：
   - feature -> `alpha-feature-dev`
   - bug -> `alpha-bug-fix`
5. 发出第一条阶段播报

## 状态字段建议

```json
{
  "runId": "R-20260317-001",
  "workflow": "alpha-feature-dev",
  "status": "running",
  "currentStage": "plan",
  "blockedReason": null,
  "approvalNeeded": false,
  "retryFrom": null
}
```

## 阶段流转规则

### intake -> plan
alpha-pilot 完成任务识别后进入 plan。

### plan -> implement
条件：
- planner 完成 requirements / design / scope
- planner 写入 handoff

### implement -> ops-verify-and-ship
条件：
- developer 完成实现摘要与变更文件清单
- developer 写入 handoff

### ops-verify-and-ship -> summary
条件：
- ops 完成 test/review/pr/merge/deploy 结果摘要
- ops 写入 handoff

### summary -> done
条件：
- alpha-pilot 已生成最终总结并播报

## 阻塞与审批

### 阻塞
当以下情况出现时，`status` 应改为 `blocked`：
- planner 需要方案拍板
- ops 验证失败且超出自动重试范围
- deploy 缺少必要配置

### 审批
当以下情况出现时，`approvalNeeded` 设为 `true`：
- 生产部署
- 高风险 merge
- 范围变更超出原计划
