# Spec: Agent Orchestration for AlphaPilot

## Background

AlphaPilot 需要从单一开发/执行方式，升级为面向长期协作的多角色 Agent 编排模式。

用户已明确收敛为四个角色：

1. `alpha-pilot`：主 Agent / 调度者
2. `planner`：分析需求、维护需求文档、收敛变动范围、输出技术方案并更新 docs
3. `developer`：基于 planner 产物开发代码 / 修复缺陷，并补齐白盒测试与单元测试
4. `ops`：执行验收、测试结果复核、部署与发布把关

同时用户明确提出以下产品要求：

- 群里只汇报阶段性结果
- 每个子角色有自己的上下文
- 避免上下文爆炸和 token 失控
- 未来会接 Telegram 多 bot，但当前先做本地改造与脚手架

## Goals

1. 定义 AlphaPilot 的四角色 Agent 组织结构
2. 定义 feature / bug 两类工作流
3. 定义 run artifacts 与 handoff 协议，避免长会话堆积
4. 定义群播报协议，只播报阶段性结果
5. 为未来 Telegram 多 bot 接入预留配置位

## Non-Goals

1. 当前阶段不直接完成 Telegram bot 真实 token 绑定
2. 当前阶段不直接实现完整 antfarm 执行层扩展代码
3. 当前阶段不替代现有业务功能开发流程

## Requirements

### Functional

#### F1. Four-role model
系统必须支持四角色模型：
- alpha-pilot
- planner
- developer
- ops

#### F2. Feature workflow
系统必须支持 feature 工作流：
- intake
- planner-analysis-and-doc-update
- implement-with-whitebox-and-unit-tests
- ops-acceptance-and-deploy
- summary

#### F3. Bug workflow
系统必须支持 bug 工作流：
- intake
- planner-analysis-and-doc-update
- fix-with-whitebox-and-unit-tests
- ops-acceptance-and-deploy
- summary

#### F4. Ops substages
ops 必须在内部支持以下子阶段：
- acceptance-check
- test-result-review
- review
- pr
- merge
- deploy

#### F4.1 Planner deliverables
planner 必须输出并维护以下文档产物：
- requirement summary
- scope / non-scope
- implementation plan
- docs updates
- developer handoff

#### F4.2 Developer quality responsibilities
developer 必须在实现阶段同时交付：
- production code
- white-box tests
- unit tests
- implementation notes / handoff

#### F5. Artifact-based state
系统必须将长期状态落地为 run artifacts，而不是依赖长上下文聊天历史。

#### F6. Handoff protocol
角色之间必须通过结构化 handoff 文件交接。

#### F7. Broadcast policy
公共群播报必须只包含阶段性结果：
- 阶段开始
- 阶段完成
- 阶段阻塞
- 请求决策
- 最终总结

### Non-functional

#### N1. Context control
默认不把群播报历史回灌到执行上下文。

#### N2. Session control
planner / developer / ops 应优先使用短会话；大任务时 developer 按 story 或子任务拆分会话。

#### N3. Extensibility
为 Telegram bot account / 群路由预留配置结构。

## Acceptance Criteria

1. 仓库内存在完整的 orchestration spec / plan / worklog 文档
2. 仓库内存在四角色职责定义与群播报协议
3. 仓库内存在 run template / handoff schema / input whitelist 文档
4. 仓库内存在 feature / bug workflow 草案
5. 文档设计与当前 superpowers workflow 一致
