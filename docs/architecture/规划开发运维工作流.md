# Planner → Developer → Ops Workflow

## Goal

把 AlphaPilot 的执行主链固定为：

1. planner 分析需求并更新 docs
2. developer 基于 planner 产物开发代码，并补齐白盒测试与单元测试
3. ops 进行验收、结果复核与部署

该流程适用于 feature 与 bugfix，两者仅在 intake 输入与验收重点上略有不同。

## Stage 0: Intake

输入来源：
- 群消息 / 私聊需求
- 缺陷反馈
- 产品变更请求
- 运行时异常与回归问题

alpha-pilot 在 intake 阶段负责：
- 抽取原始需求
- 判断是 feature / bug / ops task
- 创建 run 目录或任务上下文
- 把任务交给 planner

## Stage 1: Planner

### Responsibilities

planner 负责：
- 理解需求目标与业务动机
- 识别 scope / non-scope
- 分析影响模块、风险点、依赖项
- 更新相关 docs
- 输出可执行 implementation plan
- 生成 developer handoff

### Required outputs

进入 developer 阶段前，planner 至少要交付：
- 需求摘要
- 范围 / 非范围
- 影响面分析
- 实施步骤
- 验收标准
- 文档更新记录
- developer handoff

### Exit criteria

只有满足以下条件，任务才允许进入 developer：
- 需求目标足够明确
- 相关 docs 已更新或已明确无需更新
- 实施范围已收敛
- 存在 developer 可直接执行的 handoff

## Stage 2: Developer

### Responsibilities

developer 负责：
- 按 planner handoff 实现代码
- 补齐白盒测试
- 补齐单元测试
- 更新必要的实现文档 / 运行说明
- 产出实现结果 handoff 给 ops

### Required outputs

进入 ops 阶段前，developer 至少要交付：
- production code
- white-box tests
- unit tests
- 本地验证结果
- 已知限制 / 风险说明
- ops handoff

### Exit criteria

只有满足以下条件，任务才允许进入 ops：
- 代码已实现完成或达到可验收状态
- 白盒测试已编写
- 单元测试已编写
- 至少完成一轮本地验证
- handoff 中明确写出变更点、验证结果与剩余风险

## Stage 3: Ops

### Responsibilities

ops 负责：
- 根据 planner / developer 产物执行验收
- 复核测试结果与覆盖情况
- 做发布前 review
- 执行部署或推动合并发布
- 输出上线 / 阻塞结论

### Internal substages

ops 内部默认拆为：
1. acceptance-check
2. test-result-review
3. release-review
4. deploy
5. post-deploy-summary

### Required outputs

ops 完成后至少交付：
- 验收结论（通过 / 不通过 / 有条件通过）
- 测试复核结论
- 部署记录
- 阻塞项或回滚建议
- 最终 summary

### Exit criteria

只有满足以下条件，任务才算完成：
- 验收结论明确
- 部署结果明确
- 群内已发送阶段性总结
- 相关 run artifacts 已回填

## Broadcast policy

群里只播报：
- 阶段开始
- 阶段完成
- 阶段阻塞
- 请求决策
- 最终总结

不播报：
- 中间碎片化思考
- 冗长调试过程
- 局部试错日志

## Quality principle

质量责任前移到 developer：
- developer 不是只写功能代码
- white-box tests 与 unit tests 是实现交付的一部分
- ops 不负责替 developer 补基本测试，而是负责验收、复核和发布把关
