# Admin/Auth Overview

## Status

This document is intentionally conservative.

- The main sections below only describe facts that are both supported by `docs/认证与管理后台计划.md` / `docs/认证与管理后台规格说明.md` and directly observable in the current repository state.
- The final section records worktree observations that exist in code today but should not be treated as fully absorbed into the broader architecture narrative yet.

## Scope Boundary

The admin/auth layer sits on top of the existing AlphaPilot control plane and currently covers:

- user registration, login, and bearer-token session validation
- role and status enforcement for `user` / `admin` and `active` / `disabled`
- admin-gated writes for runtime configuration
- admin symbol configuration management
- admin user management
- audit logging for admin mutations
- frontend auth entry and admin entry surfaces

Out of scope for now:

- password reset
- 2FA
- invite-only onboarding
- multi-tenant organization support

## Backend Architecture

### Identity and Session Boundary

Backend auth is implemented in the main API router and uses a small service layer for password hashing and JWT creation/validation.

- `POST /api/auth/register` creates a normal user, hashes the password, and returns a bearer token plus user payload.
- `POST /api/auth/login` validates credentials, rejects disabled users, updates `last_login_at`, and returns a bearer token plus user payload.
- `GET /api/auth/me` resolves the current bearer token to a user record.
- `get_current_user()` extracts the bearer token, decodes it with `APP_AUTH_SECRET_KEY`, loads the user from the database, and rejects disabled users.
- `require_admin()` builds on `get_current_user()` and returns `403` for non-admin callers.

Relevant code:

- `backend/src/app/router.py`
- `backend/src/services/auth.py`

### Data Model

The auth/admin data model currently includes:

- `users`
  - fields include `username`, `email`, `password_hash`, `role`, `status`, and `last_login_at`
- `symbol_configs`
  - stores admin-maintained trading symbol metadata such as `symbol`, `base_asset`, `quote_asset`, `enabled`, `timeframe`, `max_position_size_pct`, `priority`, and `sort_order`
- `audit_logs`
  - stores admin mutation events with `action`, `resource_type`, `resource_id`, and before/after JSON payloads

Role and status enums are currently:

- `user`, `admin`
- `active`, `disabled`

Relevant code:

- `backend/src/shared/models/user.py`

- `backend/src/shared/models/audit_log.py`
- `backend/src/shared/enums.py`

### Admin API Boundary

Admin-only APIs currently visible in the router are:

- `GET /api/admin/symbols`
- `POST /api/admin/symbols`
- `PATCH /api/admin/symbols/{symbol_id}`
- `GET /api/admin/users`
- `PATCH /api/admin/users/{user_id}`
- `POST /api/config/runtime`

`GET /api/config/runtime` remains readable without the admin dependency and returns status-style fields such as configured/not-configured flags rather than plaintext secrets.

Mutation-side audit logging is directly present in the router for:

- symbol config create/update
- user role/status update

Relevant code:

- `backend/src/app/router.py`

### Default Admin Bootstrap

The backend settings model includes optional `DEFAULT_ADMIN_EMAIL`, `DEFAULT_ADMIN_PASSWORD`, and `DEFAULT_ADMIN_USERNAME`.

When email and password are provided, application startup calls `ensure_default_admin()` during FastAPI lifespan initialization. The bootstrap logic:

- creates the user if missing
- otherwise forces the account to `admin` + `active`
- syncs username and password to the configured values

This is observable in code, but it should still be treated as a dev/test bootstrap mechanism rather than a production identity model.

Relevant code:

- `backend/src/shared/config.py`
- `backend/src/services/admin_bootstrap.py`
- `backend/src/app/main.py`

## Frontend Architecture

### Auth Entry

The frontend currently includes explicit auth entry routes:

- `/login`
- `/register`

Client-side session handling is implemented with local storage and a stored bearer token.

- logout clears the stored session

Relevant code:

- `frontend/src/lib/api.ts`
- `frontend/src/lib/auth.ts`
- `frontend/src/components/auth-provider.tsx`
- `frontend/src/components/auth-card.tsx`
- `frontend/src/app/login/page.tsx`
- `frontend/src/app/register/page.tsx`

### Role-Aware App Shell

The shared app shell and route guard enforce a frontend visibility boundary, but they do not replace backend authorization.

- non-authenticated users are redirected to `/login` for protected pages
- non-admin users are blocked from admin routes in the UI
- the main navigation only shows the admin entry for admin sessions

Relevant code:

- `frontend/src/components/app-shell.tsx`
- `frontend/src/components/route-guard.tsx`

### Admin Entry Surfaces

The frontend worktree currently includes:

- `/admin`
- `/admin/currencies`
- `/admin/users`

These pages are structured as admin-only surfaces behind the shared route guard and app-shell visibility boundary.

Relevant code:

- `frontend/src/app/admin/page.tsx`
- `frontend/src/app/admin/currencies/page.tsx`
- `frontend/src/app/admin/users/page.tsx`

## Workspace Observed But Not Yet Documented

The following items are present in the current worktree and are useful to track here, but should be treated as repository observations rather than fully normalized architecture commitments:

- Frontend auth has already moved beyond a pure mock flow:
  - login/register call real backend auth endpoints
  - session refresh calls `/api/auth/me`
- Frontend admin pages already perform real admin API calls:
  - `/admin/currencies` calls `GET/POST/PATCH /api/admin/symbols`
  - `/admin/users` calls `GET/PATCH /api/admin/users`
- Frontend admin pages already include separate desktop/mobile render paths instead of relying only on wide tables.
- Backend startup already contains default-admin bootstrap logic driven by optional env settings.

These observations should be read as "implemented in the current workspace" rather than "released, deployed, or fully integrated into the broader architecture documentation set."
