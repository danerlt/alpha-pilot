# Test and Acceptance Gates

## Purpose

为 AlphaPilot 的 planner → developer → ops 主流程定义统一的准入/准出标准。

## Gate A: Planner complete

进入 developer 前必须满足：
- 需求摘要明确
- 范围与非范围明确
- 实施方案明确
- docs 已更新或注明无需更新
- 验收标准明确
- developer handoff 完整

## Gate B: Developer complete

进入 ops 前必须满足：
- 代码实现完成或达到可验收状态
- 白盒测试已提交
- 单元测试已提交
- 本地验证命令已记录
- 本地验证结果已记录
- 已知风险已记录
- ops handoff 完整

## Gate C: Ops accept / reject

任务完成前必须满足：
- 验收已执行
- 测试结果已复核
- 部署结果已记录
- 如失败，已明确 blocker 与回退建议
- 群内已发送阶段性或最终总结

## Notes on white-box tests

这里的 white-box tests 指：
- 面向内部逻辑分支的测试
- 面向错误路径 / 边界条件的测试
- 面向关键流程节点的实现级验证

它和黑盒验收不同，责任默认归 developer。

## Notes on unit tests

单元测试要求：
- 关注模块级行为
- 尽量快、稳定、可重复
- 覆盖核心逻辑与回归风险点

## Blocking policy

以下情况默认不能进入下一阶段：
- planner 没有 handoff
- developer 没有测试
- 验证结果缺失
- ops 无法判断是否可部署
