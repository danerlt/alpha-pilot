# AlphaPilot 总体架构

> 本文是 AlphaPilot 的**架构总览**，给出系统的平面划分、数据流、运行时形态与演进路线。
>
> - 深度设计以 [v1 系统设计 spec](superpowers/specs/2026-04-21-alphapilot-v1-system-design.md) 为权威；
> - 工程规范（目录/命名/分层/API 契约/异常/异步/测试/Git）以 [`docs/project.md`](project.md) 为权威；
> - 产品范围与版本规划见 [`docs/产品需求文档.md`](产品需求文档.md)。
>
> 本文只做"地图"，不重复上述文档的细节。

---

## 1. 一句话定位

AlphaPilot 是一个在**严格风控边界**内、用**结构化 AI 决策**驱动、面向 Binance 的**自主数字货币交易系统**：自动决策 → 风控守卫 → 自动执行 → 复盘沉淀 → 受控进化。

---

## 2. 四平面架构

系统按职责切成 4 个平面，**代码物理隔离、运行时可合一可拆开**（MVP 单体进程，未来任一平面可独立成进程，只换事件总线适配器，不改业务代码）。

```
┌─────────────────────────────────────────────────────────────┐
│  Control Plane（控制平面）                                  │
│  用户权限 / 审计日志 / 配置热更新 / 启停开关 / 密钥 / 消息端 │
└────────────────────┬────────────────────────────────────────┘
                     │ 配置热更新 / 手动指令事件
                     ▼
┌─────────────────────────────────────────────────────────────┐
│  Strategy Intelligence Plane（策略智能平面）                │
│  AI Trader: Prompt AI → Decision AI → Review AI             │
│  Program Trader（V0.2+）/ Shadow Mode（V0.3+）              │
│  输出统一 DecisionProposal 事件                             │
└────────────────────┬────────────────────────────────────────┘
                     │ decision.proposed / decision.reviewed
                     ▼
┌─────────────────────────────────────────────────────────────┐
│  Execution Core（执行核心）                                 │
│  Market Data / Account State / Execution Guard              │
│  Order Execution / Position Monitor                         │
│  输出 order.* / trade.* / position.* / risk.* 事件          │
└────────────────────┬────────────────────────────────────────┘
                     │ trade.closed / decision.* / risk.*
                     ▼
┌─────────────────────────────────────────────────────────────┐
│  Factor & Insight Plane（因子与洞察平面）                   │
│  Indicators / Factor Library / Regime Classifier            │
│  Attribution / Experience / Strategy Scorer / Reports       │
│  Ops & Diagnose AI（V0.3+）                                 │
└─────────────────────────────────────────────────────────────┘
```

### 平面职责

| 平面 | 职责 | 核心原则 |
|------|------|----------|
| **Control Plane** | 用户/权限/审计、运行时配置热更新、KillSwitch 启停、密钥保管（Fernet 加密，admin 不可读明文）、手动运维操作、消息端路由 | 一切关键操作可审计、可回滚、可一键停机 |
| **Strategy Intelligence** | 在受限策略框架内产出结构化决策；V0.1 单次 LLM 调用，V0.2+ 拆 Prompt→Decision→Review 多智能体 | AI 可自主但不可越权；输出必须结构化 |
| **Execution Core** | 行情/账户接入、执行守卫（PASS/REJECT/DEGRADE）、幂等下单/平仓、持仓监控与止损止盈 | 风控优先级高于 AI；下单幂等（trace_id） |
| **Factor & Insight** | 指标/因子/市场状态计算、交易归因、经验沉淀、策略评分、日报 | 学习必须先验证；硬风控不可学习 |

> ⚠️ **代码现状与本图的映射**：上图是 v1 spec 的理想平面划分。当前 `backend/src/` 的实际落地按领域子目录组织（`services/{execution,insight,strategy,risk,reporting,system}` + `controllers/api/v1/{execution,strategy,risk,system}`），平面边界体现在服务分域与事件契约上，而非顶层目录名。权威目录布局见 [`docs/project.md`](project.md) §1。

## 3. 核心数据流（V0.1 主线）

```
定时触发（scheduler，每 15 分钟）
  → Market Data 拉 K 线 → Indicators/Factors 计算 → Regime 识别
  → 检索相似经验 → Decision Pipeline（LLM 结构化决策）
  → Execution Guard 校验（PASS / REJECT / DEGRADE）
  → Order Execution 幂等下单 → 写 Position/Order/Trade
  → 发布 order.*/position.*/trade.* 事件 → EventShuttle → WS 实时推送
  → Position Monitor（每 10 秒）止损/止盈/熔断检查
  → trade.closed → Experience 沉淀 + Attribution 归因 + 日报
```

事件总线：DB `event_store`（事件溯源持久层）↔ Redis Pub/Sub（实时旁路）由 `EventShuttle` 常驻搬运。

## 4. 运行时形态

| 进程 | 入口 | 职责 |
|------|------|------|
| **api** | `scripts/start_api.py`（`src.app:app`） | FastAPI HTTP + WebSocket（多 worker） |
| **scheduler** | `scripts/start_scheduler.py` | APScheduler 定时任务 + EventShuttle daemon + 异步任务队列消费（BRPOP） |

- 数据库 PostgreSQL 16，缓存/事件/任务队列 Redis 7。
- HTTP 触发的耗时业务（一键全平、生成日报）走 `task_requests` 表异步入队 → scheduler 扫表执行 → `task.status_changed` 事件回推（三层持久化，见 [`project.md`](project.md) §8）。
- 部署：docker-compose，三环境 dev/uat/prod（CI/CD 见 [`docs/deploy-ci.md`](deploy-ci.md)）。

## 6. 关键设计约束（不可违背）

- **风控优先级高于 AI**：`ExecutionGuard` 在 LLM 决策后强制校验，失败抛 `RiskRejectedException`，不进下单链路。
- **LLM 兜底**：解析失败 / 缺 stop_loss / 非法 action → 统一回退 `HOLD`。
- **熔断不可被 LLM 覆盖**：日亏损 > 3% 或连续亏损 ≥ 3 笔 → 自动 REJECT。
- **交易模式隔离**：所有业务表带 `trading_mode`，testnet / mainnet 数据语义完全隔离。
- **下单幂等**：`Order.trace_id = SHA256(decision_id:symbol:action)`，重启可复现。
- **不可学习项**：单笔/总仓位上限、日亏损阈值、熔断规则、必须止损规则、API 异常停机——硬风控边界不参与任何学习。
